# 要件定義書

YouTube動画コメント感情分析システム

## 1. 目的

YouTube Data API v3 を利用して動画メタデータおよびコメントを取得し、3モデルBERTアンサンブル + ルールベース（200以上のパターン）による感情分析を実行し、集計結果をStreamlit WebUIまたはCLIで表示するシステムである。

本システムは分析用バッチ処理およびWebインターフェースを提供し、再実行性・多言語対応・拡張性を前提とする。

## 2. 対象範囲

- 動画メタデータ取得（再生回数、高評価数、コメント数など）
- コメント取得（いいね数順ソート対応）
- 感情分析（日本語 + 多言語対応）
- 集計・結果表示（Streamlit UI / CLI）

## 3. 機能要件

### 3.1 動画メタデータ取得

YouTube Data API v3 の `videos.list` を使用し、動画IDごとに以下のメタデータを取得する。

| 項目 | APIフィールド | 型 |
|------|-------------|-----|
| 動画ID | id | string |
| タイトル | snippet.title | string |
| チャンネルID | snippet.channelId | string |
| チャンネル名 | snippet.channelTitle | string |
| 公開日 | snippet.publishedAt | ISO 8601 |
| 再生回数 | statistics.viewCount | integer |
| 高評価数 | statistics.likeCount | integer |
| コメント数 | statistics.commentCount | integer |

### 3.2 コメント取得

- デフォルト取得件数: 10件（環境変数 `COMMENT_LIMIT` で変更可能）
- CLI引数 `--comment-limit` で実行時に指定可能
- 取得順序: relevance（関連度順）で取得後、いいね数降順にソート
- ソート最適化: `COMMENT_FETCH_MULTIPLIER`（デフォルト2倍）で多めに取得しソート

### 3.3 感情分析

3つのHugging Face公開BERTモデルによるアンサンブル + 200以上のルールベースパターンマッチングにより感情分析を実行する。

| 分類ラベル | 意味 | スコア範囲 |
|-----------|------|-----------|
| positive | ポジティブ | 0.0 - 1.0 |
| negative | ネガティブ | 0.0 - 1.0 |
| neutral | 中立/判定困難 | 0.0 - 1.0 |

- 言語検出により日本語と非日本語を自動判定
- 日本語: 2モデルアンサンブル（BERT + 皮肉検出モデル）
- 非日本語: XLM-RoBERTa多言語モデル
- ルールベース補正: 皮肉、反語、否定表現、YouTube固有表現を検出

### 3.4 集計・表示

動画単位でコメントの感情分析結果を集計し、以下を表示する。

- 感情分類別件数（positive / negative / other）
- 感情分類比率
- 平均確率スコア（positive_score, negative_score, neutral_score）
- コメント一覧（感情フィルタ機能付き）

### 3.5 Streamlit WebUI

| 機能 | 説明 |
|------|------|
| 動画ID/URL入力 | 複数形式のYouTube URL対応 |
| 分析実行 | ワンクリックで分析開始 |
| 動画情報表示 | メトリクスのカード表示 |
| 感情サマリー | 棒グラフ + スコア表示 |
| コメント一覧 | 感情フィルタ付きExpander |
| エラーハンドリング | API制限、認証エラーの日本語表示 |

対応URL形式:
- `https://www.youtube.com/watch?v=VIDEO_ID`
- `https://youtu.be/VIDEO_ID`
- `https://www.youtube.com/shorts/VIDEO_ID`
- `https://www.youtube.com/embed/VIDEO_ID`
- 11文字の動画ID直接入力

## 4. 非機能要件

### 4.1 実行環境

- Docker Compose によるコンテナ実行（Docker Desktop必須）
- Python 3.11（python:3.11-slim ベースイメージ）
- CPU専用推論（GPU不要）

### 4.2 認証・セキュリティ

- YouTube API キーは `.env` ファイルで管理（Git管理外）
- コンテナ間通信は専用ブリッジネットワーク

### 4.3 実行方式

- 手動実行（スケジューラ不使用）
- Streamlit WebUI（ブラウザからの対話的操作）
- CLIバッチ処理（Docker Compose exec経由）

```bash
# 単一動画
docker compose exec app python main.py --video-id VIDEO_ID

# コメント取得件数を指定
docker compose exec app python main.py --video-id VIDEO_ID --comment-limit 50

# 複数動画
docker compose exec app python main.py --video-ids VIDEO_ID1,VIDEO_ID2,VIDEO_ID3
```

### 4.4 データ永続化

- データの永続化（ファイル保存・DB保存）は行わない
- 実行のたびにYouTube APIからデータを取得し、分析・表示する
- ログファイルのみDockerボリュームで永続化

### 4.5 パフォーマンス要件

| 項目 | 目標値 |
|------|--------|
| 推論速度 | 50-100ms / コメント（CPU） |
| メモリ使用量 | 3.5GB以下 |
| 10コメント分析 | 約0.5-1秒 |
| 100コメント分析 | 約5-10秒 |

## 5. APIクォータ対策

### 5.1 YouTube Data API v3 のクォータ制限

- 1日あたり10,000ユニット（デフォルト）
- 主なコスト:
  - `videos.list`: 1ユニット
  - `commentThreads.list`: 1ユニット

### 5.2 対策方針

1. **リクエスト最適化**
   - 必要なフィールドのみ取得（`part` パラメータの最適化）
   - バッチリクエストで複数動画を一括取得（最大50件）

2. **エラーハンドリング**
   - クォータ超過時（403エラー）は処理を中断しログ出力
   - `QuotaExceededError` カスタム例外で明確に通知
   - リトライは翌日（太平洋時間0時リセット）まで待機

3. **実行管理**
   - 1回のバッチで処理する動画数を制限可能
   - クォータ使用量をログに記録
