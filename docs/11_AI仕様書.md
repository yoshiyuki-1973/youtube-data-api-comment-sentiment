# AI仕様書

YouTube動画コメント感情分析システム - 3-model BERTアンサンブル感情分析エンジン

## 1. 概要

本システムは3つのHugging Face公開BERTモデルによるアンサンブル + 200以上のルールベースパターンマッチングにより、YouTubeコメントの感情分析を実行する。単一モデルでは皮肉・反語を誤分類するため、アンサンブル方式を採用している。

### 1.1 使用モデル

#### 日本語モデル1: christian-phu/bert-finetuned-japanese-sentiment

| 項目 | 値 |
|------|-----|
| Hugging Face ID | `christian-phu/bert-finetuned-japanese-sentiment` |
| タイプ | 3クラス分類（negative / neutral / positive） |
| 環境変数 | `JA_MODEL_1` |
| 用途 | 日本語コメントの基本的な感情分析 |

#### 日本語モデル2: kit-nlp/bert-base-japanese-sentiment-irony

| 項目 | 値 |
|------|-----|
| Hugging Face ID | `kit-nlp/bert-base-japanese-sentiment-irony` |
| タイプ | 2クラス分類（ポジティブ / ネガティブ）+ 皮肉検出 |
| 環境変数 | `JA_MODEL_2` |
| 用途 | 日本語の皮肉表現や複雑な感情の検出 |

#### 多言語モデル: cardiffnlp/twitter-xlm-roberta-base-sentiment

| 項目 | 値 |
|------|-----|
| Hugging Face ID | `cardiffnlp/twitter-xlm-roberta-base-sentiment` |
| タイプ | 3クラス分類（negative / neutral / positive） |
| 環境変数 | `MULTILINGUAL_MODEL` |
| 用途 | 日本語以外の言語の感情分析 |
| 特徴 | Twitter特化、198言語対応 |

### 1.2 分類タスク

- **最終出力**: 3クラスの確率スコア（合計1.0）
  - `positive` (0.0-1.0): ポジティブな感情
  - `negative` (0.0-1.0): ネガティブな感情
  - `neutral` (0.0-1.0): 中立または判定困難
- **アンサンブル方式**:
  - 日本語コメント: 日本語モデル2つの予測スコア平均
  - その他言語: 多言語モデルの予測を使用
  - ルールベース: 200以上のパターンマッチングで補正

### 1.3 推論環境

| 項目 | 設定 |
|------|------|
| デバイス | CPU専用（GPU不要） |
| スレッド数 | 1（`torch.set_num_threads(1)`） |
| 最大トークン長 | 128（環境変数 `MAX_TOKEN_LENGTH` で変更可能、1-512） |
| バッチサイズ | 1（順次処理） |
| 勾配計算 | 無効化（`torch.no_grad()`） |
| メモリ使用量 | 約3.5GB以下 |

## 2. 推論フロー

### 2.1 全体フロー

```
入力テキスト
  ↓
前処理（URL除去、HTMLタグ除去、空白正規化）
  ↓
空文字判定 → 空の場合は均等分布(0.33/0.33/0.34)を返す
  ↓
言語検出
  ├── 日本語（ひらがな/カタカナ/漢字検出）
  │     ↓
  │   日本語モデル1（christian-phu: 3クラス） → スコア
  │   日本語モデル2（kit-nlp: 2クラス+皮肉検出） → スコア
  │     ↓
  │   アンサンブル（2モデルのスコア平均）
  │
  └── その他言語（langdetectフォールバック）
        ↓
      多言語モデル（XLM-RoBERTa: 3クラス） → スコア
  ↓
ルールベース補正（200+パターン）
  ↓
正規化（合計1.0）
  ↓
出力: {positive: float, negative: float, neutral: float, language: str}
```

### 2.2 前処理

```python
def _preprocess_text(text: str) -> str:
    # 1. URL除去
    text = re.sub(r'https?://\S+', '', text)
    # 2. HTMLタグ除去
    text = re.sub(r'<[^>]+>', '', text)
    # 3. 空白正規化
    text = ' '.join(text.split())
    return text.strip()
```

### 2.3 言語検出

1. **文字ベース判定**（高速）: ひらがな（\u3040-\u309F）、カタカナ（\u30A0-\u30FF）、漢字（\u4E00-\u9FFF）のいずれかを含む場合は `"ja"` を返す
2. **langdetectフォールバック**: 日本語文字がない場合は `langdetect.detect()` を使用
3. **検出失敗時**: `"other"` としてXLM-RoBERTaモデルで処理

### 2.4 単一モデル推論

```python
def _single_model_inference(text, model, tokenizer, id2label):
    inputs = tokenizer(
        text,
        padding=True,
        truncation=True,
        max_length=128,
        return_tensors='pt'
    )

    with torch.no_grad():
        outputs = model(**inputs)
        probabilities = torch.softmax(outputs.logits, dim=-1)[0]

    # 3クラスモデル: id2labelに基づきスコアをマッピング
    # 2クラスモデル: neutralは0.0として返す
    return {"positive": float, "negative": float, "neutral": float}
```

### 2.5 アンサンブル

日本語コメントの場合、2つのモデルのスコアを単純平均する:

```python
ensemble_score = {
    "positive": (model1_positive + model2_positive) / 2,
    "negative": (model1_negative + model2_negative) / 2,
    "neutral":  (model1_neutral  + model2_neutral)  / 2
}
```

kit-nlpモデルは2クラスのため `neutral=0.0` となり、アンサンブル後のneutralスコアは主にchristian-phuモデルの値に基づく。

## 3. ラベルマッピング

### 3.1 モデル出力の正規化

異なるモデルが異なるラベル形式を返すため、統一的にマッピングする:

| モデル出力 | 正規化後 |
|-----------|---------|
| `pos`, `positive`, `LABEL_1`, `1` | `positive` |
| `neg`, `negative`, `LABEL_0`, `0` | `negative` |
| `other`, `neutral`, `LABEL_2`, `2` | `neutral` |

### 3.2 2クラスモデル（kit-nlp）のマッピング

| id2label | 出力 |
|---------|------|
| `{0: 'ポジティブ', 1: 'ネガティブ'}` | positive=scores[0], negative=scores[1], neutral=0.0 |

## 4. ルールベースエンジン

### 4.1 パターン一覧

| 定数名 | パターン数 | 用途 |
|--------|-----------|------|
| `POSITIVE_WORDS` | 50+ | ポジティブキーワード（日英+絵文字） |
| `NEGATIVE_WORDS` | 50+ | ネガティブキーワード（日英+絵文字） |
| `STRONG_NEGATIVE_PATTERNS` | 100+ | 強いネガティブ表現（YouTube特有含む） |
| `STRONG_POSITIVE_PATTERNS` | 100+ | 強いポジティブ表現 |
| `SARCASM_PATTERNS` | 20+ | 皮肉・棒読み検出（正規表現） |
| `RHETORICAL_PATTERNS` | 20+ | 反語表現検出（正規表現） |
| `NEGATION_PATTERNS` | 15+ | 否定表現（感情反転用） |

### 4.2 ルールベース分類（_rule_based_classify）

モデル推論が使用できない場合のフォールバック。`POSITIVE_WORDS` と `NEGATIVE_WORDS` のマッチ数を比較して `"pos"` または `"neg"` を返す。同数の場合は `"pos"` を返す。

### 4.3 ルールベース補正（_adjust_sentiment_with_rules）

モデル推論後のスコアに対する補正。以下の順序で適用:

| 優先度 | 条件 | 補正 |
|--------|------|------|
| 1 | 強いネガティブ表現 x 2以上 | negative +0.25, positive -0.25 |
| 1 | 強いネガティブ表現 x 1 | negative +0.15, positive -0.15 |
| 2 | 皮肉表現（棒読み・婉曲等） | negative +0.2, positive -0.2 |
| 3 | 反語表現（「これが面白い？」等） | negative +0.2, positive -0.2 |
| 4 | ポジティブ + 否定（「面白くない」） | negative +0.2, positive -0.2 |
| 5 | 強いポジティブ表現 x 2以上（否定なし） | positive +0.25, negative -0.25 |
| 5 | 強いポジティブ表現 x 1（否定なし） | positive +0.15, negative -0.15 |

**補正値の制限**: 各方向の累積補正は最大 +/-0.3 に制限される。
**正規化**: 補正後、スコアの合計が1.0になるよう正規化する。

## 5. フォールバック戦略

### 5.1 モデルロード失敗時

各モデルのロード失敗は個別に処理され、残りのモデルは継続してロードされる。

| 状態 | 動作 |
|------|------|
| 3モデルとも成功 | 通常のアンサンブル推論 |
| 日本語モデル1のみ成功 | 日本語モデル1のみで推論 |
| 日本語モデル2のみ成功 | 日本語モデル2のみで推論 |
| 全モデル失敗 + FALLBACK_TO_RULES_ONLY=true | ルールベースのみで分類 |
| 全モデル失敗 + FALLBACK_TO_RULES_ONLY=false | RuntimeErrorを送出 |

### 5.2 ルールベースフォールバックのスコア

| 判定結果 | positive | negative | neutral |
|---------|---------|---------|---------|
| pos | 0.6 | 0.15 | 0.25 |
| neg | 0.15 | 0.6 | 0.25 |

フォールバック後もルール補正（_adjust_sentiment_with_rules）が適用される。

### 5.3 モデルロード戦略

```python
# グローバル変数で管理（起動時に1回のみロード）

        return  # Already loaded

    with _model_lock:
        if _ja_model_1 is not None and ...:
            return  # Already loaded by another thread

        # 3モデルを順次ロード
        # 個別の失敗はログ出力し、残りを継続
```

## 8. 制約事項

### 8.1 技術的制約

- **GPU不使用**: CPUのみで動作
- **並列処理禁止**: メモリ効率優先で順次処理
- **トークン長制限**: 128トークン（約85-100文字程度）を超える部分は切り詰め
- **バッチ処理なし**: 1コメントずつ処理

### 8.2 分類の制約

- **3値分類**: positive/negative/neutralの確率スコアを出力
- **日本語特化**: 日本語以外はXLM-RoBERTaで対応（精度は日本語モデルより低い場合がある）
- **短文に最適化**: 長文は128トークンに切り詰められる
